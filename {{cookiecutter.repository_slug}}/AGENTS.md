# AGENTS.md

## Назначение репозитория

Этот репозиторий является primary source of truth для архитектуры, требований, контрактов, фич и возможностей продукта, тестовых сценариев, release notes и документации одного продукта или одного проекта.

Код продукта в этом репозитории не хранится.

## Основной принцип

Работа ведётся по подходу Architecture / Documentation First.

Любая существенная фича или изменение поведения системы сначала описывается в архитектурном репозитории, согласуется через MR/PR, затем реализуется в кодовых репозиториях.

MR/PR в архитектурный репозиторий мержится только после того, как реализация в кодовых репозиториях завершена и проверена на соответствие документации.

## Конфигурация стандарта

Локальный репозиторий должен оставаться самодостаточным.

Для этого используются:

- `arch-docs.yaml` — машиночитаемая конфигурация локального стандарта;
- `.arch-docs/standard/manifest.yaml` — обязательные файлы, роли, каталоги и этапы процесса;
- `.arch-docs/standard/migrations.yaml` — цепочка миграций локального стандарта;
- `.arch-docs/wiki-llm/index-manifest.yaml` — детерминированный индекс документации для retrieval.

Изменения cookiecutter-стандарта должны приезжать в локальный репозиторий через `audit` и `migrate`, а не через ручное редактирование без проверки версии.
Реальное обновление файлов стандарта должно вестись через skill `.arch-docs/agents/standard-migrator`, а не сводиться к одному переключению версии в конфиге.

## Язык документации

Вся документация ведётся на русском языке.

## Branch model

- `{{cookiecutter.default_branch}}` — актуальное production-состояние документации.
- `{{cookiecutter.feature_prefix}}/*` — будущие изменения.
- `release/*` — состояние конкретных релизов.
- `tags` — зафиксированные release snapshots.

## Workflow разработки фичи

1. Получить описание фичи.
2. Прочитать `arch-docs.yaml`.
3. Проверить актуальность `.arch-docs/wiki-llm/index-manifest.yaml`.
4. Прочитать `repository.yaml`.
5. Прочитать `overview.md` и `glossary.md`.
6. Прочитать текущую профильную документацию.
7. Изучить доступные кодовые репозитории, если доступ есть.
8. Задать пользователю уточняющие вопросы.
9. Не придумывать неизвестные требования.
10. Создать feature branch по шаблону `{{cookiecutter.feature_prefix}}/FEATURE-XXX-short-name`.
11. Создать или обновить файл `features/FEATURE-XXX-short-name.md`.
12. Разложить изменения по профильным разделам документации.
13. Запустить review skills до коммита.
14. Исправить критичные замечания.
15. Создать commit.
16. Создать MR/PR.
17. Дождаться human review.
18. После реализации в кодовых репозиториях проверить соответствие реализации документации.
19. Если реализация соответствует документации, MR/PR может быть смержен в `{{cookiecutter.default_branch}}`.
20. Если фичу решили не делать, MR/PR закрывается без merge.

## Workflow первичного заполнения нового репозитория

Если репозиторий только что сгенерирован и системный контур ещё не заполнен, агент должен работать в interview-first режиме.

Правила режима:

1. Сначала прочитать `arch-docs.yaml`, `repository.yaml`, `overview.md` и `glossary.md`.
2. Определить, какие обязательные поля и разделы ещё пусты или заполнены placeholder-значениями.
3. Задавать пользователю по одному вопросу за раз, а не выгружать большой опросник целиком.
4. После каждого ответа сразу обновлять соответствующие файлы, а не откладывать все правки на конец.
5. Если ответ неизвестен, фиксировать `Неизвестно / требует уточнения` и переходить дальше.
6. Сначала закрыть минимальный системный контур в `repository.yaml`, `overview.md`, `glossary.md`, `architecture/requirements/`, `architecture/api/`, `architecture/data/`, `architecture/deployment/`.
7. После этого пройти по placeholder-значениям и `unknown`, которые уже можно заменить на основе диалога с пользователем.
8. Для placeholder-значений также задавать по одному вопросу за раз, группируя только явно связанные пункты.
9. После этого запустить базовые проверки репозитория и разобрать результаты.
10. По каждой ошибке сначала попытаться исправить проблему самостоятельно, а вопросы пользователю задавать только там, где без уточнения нельзя принять решение.
11. После исправлений повторно запускать проверки, пока репозиторий не будет доведён до рабочего состояния или пока не останутся только блокеры из-за отсутствующих входных данных.
12. После каждой серии ответов и проверок кратко показывать, что уже заполнено, что исправлено и какие вопросы ещё остались.

Минимальный порядок интервью:

1. Название продукта, `project_id`, владелец и границы системы.
2. Репозитории продукта.
3. Сервисы и компоненты.
4. Хранилища данных.
5. Внешние системы и интеграции.
6. Окружения.
7. Роли принятия продуктовых и архитектурных решений.

## Локальные режимы репозитория

### `audit`

Проверяет:

- не отстал ли локальный стандарт от `target_version`;
- не отсутствуют ли обязательные файлы, роли, каталоги и скрипты;
- не устарел ли `.arch-docs/wiki-llm/index-manifest.yaml`;
- не остались ли template placeholders и незаполненные участки.

Команда:

- `./.arch-docs/scripts/audit-standard.sh --fail-on-gap`

### `migrate`

Используется после обновления шаблона в репозитории.

Задача режима:

- прочитать `.arch-docs/standard/migrations.yaml`;
- построить цепочку миграций до `target_version`;
- перед применением сравнить cookiecutter-репозиторий и локальный репозиторий по артефактам стандарта;
- применить её шаг за шагом через skill `.arch-docs/agents/standard-migrator`;
- при необходимости сделать commit на каждый migration step.

Команды:

- `./.arch-docs/scripts/migrate-standard.sh`
- `./.arch-docs/scripts/migrate-standard.sh --apply --commit`

### `bootstrap`

Используется для существующего проекта, который нужно собрать в arch-docs repo as is.

Режим должен:

- собрать факты из текущей документации, YAML, схем, CI-конфигов и доступных артефактов;
- сформировать gap-report;
- разложить текущее состояние по разделам;
- пометить всё неполное как `Неизвестно / требует уточнения`.

Команда:

- `./.arch-docs/scripts/bootstrap-repo.sh --write-report`

### `wiki-llm sync`

`wiki-llm` обязателен для локального workflow.

После любого изменения документации нужно:

1. Обновить индекс:
   `./.arch-docs/scripts/wiki-llm-sync.sh`
2. Проверить audit:
   `./.arch-docs/scripts/audit-standard.sh --fail-on-gap`

## Правила features/

В `features/` хранятся канонические описания реальных фич и возможностей продукта/системы.

Формат файла:

`features/FEATURE-XXX-short-name.md`

В `{{cookiecutter.default_branch}}` не должно быть draft-фич.

Для каждой фичи нужно явно фиксировать её process impact в секции `Затронутые бизнес-процессы`.
Feature-файл является точкой входа в документацию по этой возможности, а профильные разделы раскрывают детали по своим доменам.

## Правила обновления документации

- Файл в `features/` является единым описанием фичи или возможности системы, а не контейнером для вспомогательных артефактов.
- Если изменение меняет поведение системы, сначала обновляется соответствующий feature-файл; чисто редакторские или стандартные правки без изменения поведения могут идти без создания отдельной synthetic feature.
- Неизвестные факты фиксируются как `Неизвестно / требует уточнения`.
- При наличии impact изменения должны быть разложены по `architecture/`, включая `architecture/requirements/`, `architecture/api/`, `architecture/data/`, `architecture/testing/`, `architecture/deployment/`, а также по `release-notes/`.
- При наличии архитектурного выбора нужно создать или обновить ADR.

## Правила CI

CI выполняет только детерминированные проверки.

CI не должен вызывать LLM, Codex skills, агентов или внешние reasoning-сервисы.

## Обязательные локальные review skills

Перед коммитом запустить:

- `standard-migrator`, если в ветке меняется версия стандарта или служебные артефакты шаблона;
- `analyst-architect` self-check;
- `documentation-mapper`;
- `completeness-reviewer`;
- `architecture-reviewer`;
- `contract-reviewer`;
- `test-reviewer`;
- `security-reviewer`.

## Справочник по roles

Ниже приведено краткое назначение каждого skill из `.arch-docs/agents/*/SKILL.md`.

- `.arch-docs/agents/standard-migrator`:
  Используется при обновлении локального репозитория на новую версию cookiecutter-стандарта.
  Сравнивает шаблон и локальный репозиторий, выделяет локальные кастомизации, затем проводит миграцию по `.arch-docs/standard/migrations.yaml`.

- `.arch-docs/agents/analyst-architect`:
  Используется в начале работы над новой или изменяемой фичей.
  Собирает уточнения, формирует draft feature-файла, определяет impacted sections и решает, нужен ли ADR.

- `.arch-docs/agents/documentation-mapper`:
  Используется после согласования feature-инкремента.
  Раскладывает изменения по профильным разделам документации и проверяет непротиворечивость ссылок между feature, tests, contracts и release notes.

- `.arch-docs/agents/completeness-reviewer`:
  Используется перед коммитом как проверка полноты.
  Ищет пропуски в problem statement, requirements, acceptance criteria, e2e, release notes, deployment impact и traceability.

- `.arch-docs/agents/architecture-reviewer`:
  Используется для архитектурного review документационного инкремента.
  Проверяет целостность между `overview.md`, system boundaries, container architecture, LLD, integrations, security, observability и ADR.

- `.arch-docs/agents/contract-reviewer`:
  Используется при изменениях API-контрактов, event-контрактов или data contracts.
  Проверяет backward compatibility, breaking changes, схемы payload и наличие миграционного описания.

- `.arch-docs/agents/test-reviewer`:
  Используется для проверки тестового покрытия.
  Сопоставляет acceptance criteria с e2e, contract, security, load и regression-покрытием.

- `.arch-docs/agents/security-reviewer`:
  Используется при наличии security или privacy impact.
  Проверяет authentication, authorization, PII, audit, secrets, trust boundaries, внешние интеграции и обязательные security tests.

- `.arch-docs/agents/implementation-verifier`:
  Используется после реализации в кодовых репозиториях.
  Сверяет feature и impacted docs с реальными MR/PR и формирует verification report по соответствию документации и реализации.

## Правила создания MR/PR

Описание MR/PR должно содержать:

- Feature ID;
- краткое описание фичи;
- ссылку на задачу;
- список изменённых разделов документации;
- открытые вопросы;
- результаты review skills;
- чеклист готовности к implementation;
- явное указание `Не мержить до завершения реализации в кодовых репозиториях`.

## Как вызывать роли в Codex

Вызывать роли нужно явно, текстом в запросе.

Базовый паттерн:

1. Сказать `Работай по AGENTS.md`.
2. При необходимости попросить выполнить `audit`, `migrate`, `bootstrap` или `wiki-llm sync`.
3. Для обновления стандарта сначала назвать skill `.arch-docs/agents/standard-migrator`.
4. Для предметных изменений назвать один или несколько skills из `.arch-docs/agents/*/SKILL.md`.
5. Описать задачу и ожидаемый артефакт.

Примеры команд:

- `Codex, работай по AGENTS.md и используй skill .arch-docs/agents/analyst-architect. Сначала задай уточняющие вопросы по новой фиче.`
- `Codex, работай по AGENTS.md. Помоги заполнить системный контур нового продукта: задавай по одному вопросу и после каждого ответа обновляй документы.`
- `Codex, работай по AGENTS.md. Сначала проверь audit локального стандарта и скажи, есть ли gap относительно target_version.`
- `Codex, работай по AGENTS.md. Сначала проверь, актуален ли wiki-llm индекс. Если нет, выполни wiki-llm sync, затем прогони audit и продолжай работу.`
- `Codex, работай по AGENTS.md и используй skill .arch-docs/agents/standard-migrator. Сравни cookiecutter-репозиторий и текущий репозиторий, затем выполни миграцию локального стандарта на target_version.`
- `Codex, работай по AGENTS.md. Подготовь bootstrap gap-report для существующего проекта as is.`
- `Codex, работай по AGENTS.md и используй skill .arch-docs/agents/documentation-mapper. Разложи согласованные изменения по профильным разделам документации.`
- `Codex, работай по AGENTS.md и используй skills .arch-docs/agents/completeness-reviewer, .arch-docs/agents/contract-reviewer, .arch-docs/agents/test-reviewer. Проверь текущую ветку и сформируй review report.`

Правило шаблона:

- примеры вызова и сценарии работы держим в `AGENTS.md` и `.arch-docs/agents/*/SKILL.md`;
- предметные документы репозитория не должны содержать демо-фичу, демо-домен или встроенный walkthrough.

## Правила проверки реализации

После merge кода нужно:

1. Прочитать feature-файл.
2. Сопоставить acceptance criteria и реализацию.
3. Проверить API, data, testing и deployment sections.
4. Сверить фактические MR/PR в кодовых репозиториях.
5. Зафиксировать статусы:
   - соответствует;
   - не соответствует;
   - невозможно проверить;
   - требует ручной проверки.

## Критичные ошибки

MR/PR не должен быть создан или должен быть помечен как not ready, если есть:

- конфликт с текущей документацией;
- неполное описание фичи;
- отсутствие acceptance criteria;
- отсутствие e2e тест-кейсов;
- отсутствие описания API/event/data impact при наличии такого impact;
- отсутствие ADR при наличии архитектурного выбора;
- отсутствие release notes;
- отсутствие описания runtime/deployment impact в `architecture/deployment/` при наличии такого impact.
