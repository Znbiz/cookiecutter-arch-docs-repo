# standard-migrator

## Назначение

Вести миграцию локального arch-docs репозитория на новую версию стандарта через анализ изменений в cookiecutter-репозитории и текущем локальном репозитории.

## Когда использовать

- локальный репозиторий отстал от `arch-docs.yaml.standard.target_version`;
- cookiecutter-стандарт изменился и нужно перенести изменения в уже живой репозиторий;
- требуется не только показать diff, но и реально обновить файлы, структуру и служебную конфигурацию;
- нужно понять, какие локальные кастомизации нельзя перезатирать автоматически.

## Как вызвать из Codex

`Codex, работай по AGENTS.md и используй skill .arch-docs/agents/standard-migrator. Сначала проанализируй изменения стандарта относительно локального репозитория, затем выполни миграцию и перечисли конфликтные места.`

## Обязательный вход

- путь или доступный checkout cookiecutter-репозитория со стандартом;
- локальный arch-docs репозиторий, который нужно мигрировать;
- при наличии: целевая версия стандарта или диапазон версий.

## Перед началом

Прочитать:

1. `README.md`
2. `AGENTS.md`
3. `arch-docs.yaml`
4. `.arch-docs/standard/manifest.yaml`
5. `.arch-docs/standard/migrations.yaml`
6. `.docs/codex-workflow.md`
7. `repository.yaml`
8. `git status`

## Обязательный алгоритм

1. Проверить `applied_version`, `target_version` и состояние worktree.
2. Выполнить локальный `audit` и зафиксировать gap-report.
3. Сравнить cookiecutter-репозиторий и локальный репозиторий по служебным артефактам стандарта.
4. Отделить стандартные изменения от локальных кастомизаций.
5. Построить миграционный план по шагам из `.arch-docs/standard/migrations.yaml`.
6. Перед редактированием перечислить, какие файлы будут добавлены, перемещены, удалены или переписаны.
7. Выполнить миграцию во всех затронутых файлах, не теряя локальные продуктовые данные.
8. Обновить `arch-docs.yaml`, `wiki-llm` manifest и служебные ссылки.
9. Повторно запустить `audit` и убедиться, что стандарт консистентен.
10. Если есть конфликты или неоднозначные кастомизации, остановиться и задать пользователю точный вопрос до продолжения.

## Правила работы

- Не ограничиваться обновлением одной версии в `arch-docs.yaml`.
- Не считать `migrate-standard.sh --apply` достаточной миграцией без фактического обновления файлов стандарта.
- Не перетирать локальные продуктовые документы без явной причины.
- Если локальный файл осознанно отклоняется от шаблона, зафиксировать это как локальную кастомизацию.
- Если cookiecutter-репозиторий недоступен, явно сказать, что миграция выполняется только по локальному migration index и текущему состоянию репозитория.
- При необходимости делать commit на каждый migration step, если этого требует локальный workflow.

## Выход

- summary отличий между стандартом и локальным репозиторием;
- список изменённых файлов;
- список сохранённых локальных кастомизаций;
- список конфликтов или мест, требующих ручного решения;
- подтверждение, что `audit` после миграции проходит, либо перечень оставшихся gap.

## Критерии качества

- миграция обновляет все обязательные файлы стандарта, а не только версию;
- локальные продуктовые артефакты не повреждены;
- после миграции ссылки и пути консистентны;
- явно отделены стандартные изменения от кастомных.
