# Руководство по внедрению и работе с arch-docs репозиторием

## Что это за инструмент

Этот шаблон создаёт отдельный архитектурно-документационный репозиторий для подхода Architecture / Documentation First.

Его задача не в том, чтобы хранить код. Его задача в том, чтобы быть primary source of truth для:

- требований;
- фич и изменений продукта;
- архитектурных решений;
- API и event-контрактов;
- модели данных;
- тестовых сценариев;
- deployment/runtime-решений;
- release notes;
- локального workflow работы с Codex.

Идея простая: сначала команда договаривается о том, что именно строится и как это должно работать, потом реализует это в кодовых репозиториях, а затем подтверждает, что реализация соответствует документации.

## Главный принцип работы

Пользователь не должен вручную ходить по файлам и править документацию вслепую.

Нормальный режим работы такой:

1. Пользователь выбирает задачу: bootstrap нового репозитория, описание фичи, раскладка impact, review, migration или verification.
2. Пользователь явно вызывает нужного агента или просит работать по `AGENTS.md`.
3. Агент сам читает контекст, задаёт вопросы, обновляет документы и запускает нужные проверки.
4. Пользователь в основном отвечает на вопросы, принимает решения и валидирует итог.

Ручное редактирование допустимо, но не должно быть основным operating model.

## Почему инструмент устроен именно так

### Отдельный репозиторий, а не папка внутри кода

Отдельный репозиторий нужен, чтобы документация не была побочным артефактом команды разработки.

Если архитектурные и продуктовые решения лежат рядом с кодом одного сервиса, почти всегда происходит одно из двух:

1. документация начинает описывать только локальную часть системы;
2. документация устаревает, потому что никто не отвечает за целостную картину.

Выделенный репозиторий заставляет вести разговор о системе целиком: о границах продукта, контрактах, данных, интеграциях, тестовой стратегии и процессе вывода изменений в production.

### YAML плюс Markdown, а не только текст

В шаблоне сочетаются narrative-документы и машиночитаемые описания.

- Markdown нужен для объяснения решений, ограничений, компромиссов и контекста.
- YAML нужен для структурированных сущностей, которые можно валидировать, индексировать и использовать в automation.

Из-за этого `repository.yaml`, OpenAPI, AsyncAPI и schema-файлы не являются приложением к документации. Они часть самой документации.

### Детерминированные скрипты и проверки, а не “умный CI”

В CI здесь намеренно нет агентного reasoning и semantic review.

Причина в том, что production-ветка документации должна проверяться воспроизводимо:

- ссылки либо валидны, либо нет;
- схемы либо проходят валидацию, либо нет;
- обязательные файлы либо существуют, либо нет;
- индекс retrieval либо синхронизирован, либо нет.

Semantic-разбор и review выполняются локально через Codex skills до коммита, а не в CI после пуша.

### Local skills, а не универсальный один промпт

В `.arch-docs/agents/` роли разложены по задачам:

- `analyst-architect` помогает проработать новую фичу;
- `documentation-mapper` раскладывает impact по разделам;
- review-roles ищут дыры в полноте, контрактах, тестах, безопасности и архитектуре;
- `implementation-verifier` сверяет документацию с реализацией;
- `standard-migrator` ведёт обновление самого стандарта репозитория.

Такое разделение нужно, чтобы разные режимы работы не смешивались в один размытый процесс.

### Audit, bootstrap и migrate как отдельные режимы

У репозитория есть три разных состояния, и шаблон это признаёт явно:

- `audit` проверяет соответствие локального репозитория стандарту;
- `bootstrap` помогает поднять документацию для уже существующего продукта;
- `migrate` помогает обновлять сам стандарт репозитория без ручного хаоса.

Это лучше, чем один “setup script”, потому что новый продукт, зрелый legacy-проект и обновление стандарта решают разные задачи.

## Когда этот инструмент подходит лучше всего

Инструмент особенно полезен, если:

- продукт состоит из нескольких сервисов или репозиториев;
- есть заметный API/event/data impact между командами;
- важны traceability и согласование изменений до реализации;
- нужно удерживать единый словарь, модель системы и контракты;
- команда хочет использовать Codex не как чат, а как управляемый documentation workflow.

Если у вас один маленький сервис без интеграций и без регулярных кросс-командных изменений, такой уровень формализации может быть избыточен.

## Как начинать, если разработка только стартует

### 1. Сгенерировать репозиторий из шаблона

Создайте репозиторий через `cookiecutter`, затем сразу определите:

- стабильный `project_id`;
- человекочитаемое имя продукта;
- `repository_slug`;
- команду-владельца;
- основной git provider;
- default branch.

На этом этапе не пытайтесь подробно заполнять все разделы. Сначала задайте каркас.

### 2. Заполнить системный контур продукта

Сразу после генерации первым делом приведите в порядок:

- `repository.yaml`;
- `overview.md`;
- `glossary.md`;
- ключевые разделы `architecture/requirements/`;
- базовые данные по `architecture/api/`, `architecture/data/`, `architecture/deployment/`.

Эту часть лучше заполнять не вручную списком, а в диалоге с агентом.

Правильный режим работы:

- агент задаёт по одному вопросу за раз;
- после каждого ответа сразу обновляет соответствующие документы;
- если данных пока нет, фиксирует `Неизвестно / требует уточнения`;
- не перескакивает к следующему крупному блоку, пока не закрыл текущий минимум;
- в конце показывает, что уже заполнено и какие вопросы ещё остались.

Минимум, который должен быть известен в начале:

- какие репозитории будут у продукта;
- какие сервисы/компоненты планируются;
- какие хранилища данных предполагаются;
- какие внешние системы будут интегрированы;
- какие среды будут использоваться;
- кто принимает продуктовые и архитектурные решения.

Рекомендуемый порядок вопросов:

1. Название продукта, `project_id`, владелец и границы системы.
2. Какие репозитории входят в продукт и зачем нужен каждый.
3. Какие сервисы или компоненты планируются.
4. Какие хранилища данных нужны и кто ими владеет.
5. Какие внешние системы и интеграции ожидаются.
6. Какие окружения нужны на старте.
7. Кто принимает продуктовые, архитектурные и эксплуатационные решения.

Хороший запрос к агенту:

`Codex, работай по AGENTS.md. Помоги заполнить системный контур нового продукта. Задавай по одному вопросу, после каждого моего ответа обновляй repository.yaml, overview.md, glossary.md и связанные разделы. Неизвестное помечай как "Неизвестно / требует уточнения".`

### 3. Убрать шаблонные placeholder-значения

До активной работы нужно заменить `unknown`, `Опишите ...`, `Замените ...` и cookiecutter placeholders там, где ответ уже известен.

Это тоже лучше делать в диалоге с агентом, а не ручным поиском по всем файлам.

Правильный режим работы:

- агент находит незаполненные места;
- группирует их по смыслу, чтобы не задавать лишние повторяющиеся вопросы;
- задаёт по одному уточняющему вопросу за раз;
- после ответа сразу заменяет placeholder в нужных файлах;
- если ответа нет, ставит `Неизвестно / требует уточнения`.

Если ответ пока неизвестен, не выдумывайте. Фиксируйте это явно как `Неизвестно / требует уточнения`.

Хороший запрос к агенту:

`Codex, работай по AGENTS.md. Найди все known placeholders и unknown-значения, которые уже можно заполнить. Задавай по одному вопросу, после каждого ответа сразу обновляй документы, а неизвестное помечай как "Неизвестно / требует уточнения".`

### 4. Прогнать базовые проверки

После первичного заполнения это тоже лучше делать через агента.

Правильный режим работы:

- агент сам запускает базовые проверки;
- показывает, какие проверки прошли, а какие нет;
- кратко объясняет причину каждого fail;
- предлагает или сразу вносит правки в документацию;
- повторно запускает проверки после исправлений;
- останавливается, когда репозиторий доведён до рабочего состояния или когда упирается в отсутствующие данные.

Типовой цикл:

1. Запустить базовые проверки.
2. Разобрать ошибки и gaps.
3. Задать пользователю только те вопросы, без которых нельзя исправить результат.
4. Обновить документы.
5. Повторно прогнать проверки.

Базовые команды, которые агент должен использовать:

```bash
./.arch-docs/scripts/audit-standard.sh --fail-on-gap
./.arch-docs/scripts/wiki-llm-sync.sh
./.arch-docs/scripts/validate-docs.sh
./.arch-docs/scripts/validate-links.sh
./.arch-docs/scripts/validate-schemas.sh
./.arch-docs/scripts/validate-mermaid.sh
./.arch-docs/scripts/validate-openapi.sh
./.arch-docs/scripts/validate-asyncapi.sh
```

Задача этого шага не “сделать идеально”, а быстро довести репозиторий до рабочего состояния без структурных дыр.

Хороший запрос к агенту:

`Codex, работай по AGENTS.md. Запусти базовые проверки после первичного заполнения, сам разбирай ошибки по очереди, задавай уточняющие вопросы только там, где без них нельзя, после исправлений перезапускай проверки и доведи репозиторий до рабочего состояния.`

### 5. Первую продуктовую работу начинать не с API, а с фичи

Для нового продукта главный рабочий цикл лучше строить так:

1. описать capability или feature;
2. уточнить impact на процессы и требования;
3. разложить impact по архитектуре, контрактам, данным, тестам и deployment;
4. только после этого синхронизировать кодовые репозитории.

Именно поэтому feature-файл считается каноническим описанием изменения, а остальные разделы уточняют его со своей стороны.

Для этого шага лучше явно использовать агента, который помогает описывать фичу через диалог.

Рекомендуемый агент:

- `.arch-docs/agents/analyst-architect` из `AGENTS.md`.

Что он должен делать:

- собрать недостающий контекст по фиче;
- задать уточняющие вопросы по одному блоку за раз;
- сформировать draft feature-файла;
- определить, какие разделы документации затронуты;
- подсказать, нужен ли ADR, API-contract change, data impact, deployment impact и новые тесты.

Как вызвать:

`Codex, работай по AGENTS.md и используй skill .arch-docs/agents/analyst-architect. Помоги описать новую фичу: сначала задай уточняющие вопросы, потом подготовь draft feature-файла и список затронутых разделов документации.`

Как строить диалог:

1. Сначала дать агенту короткое описание идеи, проблемы или capability.
2. Дождаться серии уточняющих вопросов, а не пытаться сразу написать полный spec.
3. Отвечать конкретно: кто пользователь, какой сценарий меняется, что считается успешным результатом, какие есть ограничения.
4. Если что-то неизвестно, явно говорить об этом, а не придумывать ответ.
5. После первой серии ответов попросить показать draft feature и список открытых вопросов.
6. После согласования draft попросить разложить impact по `architecture/`, `api/`, `data/`, `testing/`, `deployment/` и `release-notes/`.

Хороший старт диалога:

`Нужно описать новую фичу: клиент может подписаться на уведомления о смене статуса заказа. Работай по AGENTS.md и используй skill .arch-docs/agents/analyst-architect. Сначала собери уточнения, потом подготовь draft feature-файла.`

## Как начинать, если продукт уже существует

### 1. Не пытаться сразу “задокументировать всё”

Для существующего проекта основная ошибка состоит в том, что команда пытается за один проход полностью восстановить всю документацию.

Это почти всегда заканчивается тем, что репозиторий либо остаётся полупустым, либо становится набором недостоверных текстов.

Правильная цель на старте другая: собрать as-is картину достаточно хорошо, чтобы дальше можно было работать итеративно.

Для этого тоже лучше не идти по файлам вручную, а запускать agent-driven bootstrap.

### 2. Сначала описать inventory системы

Для legacy или уже работающего продукта в первую очередь нужно заполнить:

- список кодовых репозиториев;
- каталог сервисов;
- datastore ownership;
- внешние интеграции;
- среды;
- основные контракты;
- роли согласования.

Это даёт карту системы. Без неё любые разделы про требования, тесты и deployment будут неполными.

Этот inventory должен собираться агентом: он сам читает доступные артефакты, показывает gaps и задаёт вопросы только там, где факты нельзя восстановить автоматически.

### 3. Использовать bootstrap-режим

Для первичного подъёма существующего продукта нужен режим `bootstrap`.

Лучше запускать его через агента, а не как отдельный ручной ритуал.

Что должен делать агент:

- запустить `bootstrap`;
- собрать факты из доступных файлов и конфигов;
- сформировать `bootstrap-gap-report`;
- заполнить найденный inventory по мере подтверждения фактов;
- задавать пользователю точечные вопросы по ownership, интеграциям, окружениям и критичным бизнес-флоу;
- после правок прогнать базовую валидацию результата.

Базовая команда bootstrap, которую агент должен использовать:

```bash
./.arch-docs/scripts/bootstrap-repo.sh --write-report
```

Дальше агент должен разбирать `bootstrap-gap-report` и дозаполнять документацию по приоритету:

- сначала системный inventory;
- затем критичные бизнес-флоу и контракты;
- затем тестовая и эксплуатационная часть;
- затем второстепенные разделы.

Хороший запрос к агенту:

`Codex, работай по AGENTS.md. Подними arch-docs для существующего продукта as is: запусти bootstrap, собери inventory из доступных артефактов, покажи gap-report, затем задавай уточняющие вопросы по одному блоку и после каждого ответа обновляй документы и повторно валидируй результат.`

### 4. Источники правды нужно собирать из нескольких мест

Для существующего продукта данные обычно уже размазаны по разным источникам:

- README кодовых репозиториев;
- CI-конфиги;
- OpenAPI/AsyncAPI/spec файлы;
- схемы БД;
- runbooks;
- существующие wiki/markdown документы;
- знания команды.

Репозиторий arch-docs не должен притворяться, что всё это уже было хорошо структурировано. Его задача как раз в том, чтобы собрать это в управляемую модель.

Именно поэтому пользователь не должен сам вручную переносить всё это в arch-docs. Эту работу должен делать агент, а пользователь должен подтверждать или уточнять спорные места.

Как это запускать:

1. Сказать агенту, что нужно поднять документацию для существующего продукта.
2. Попросить его работать по `AGENTS.md` и выполнить `bootstrap`.
3. Если известны источники, перечислить их прямо в запросе: какие репозитории смотреть, где лежат spec-файлы, runbooks или wiki.
4. Дальше агент сам читает доступные артефакты, собирает факты, показывает gaps и задаёт вопросы только по спорным местам.

Хороший запрос к агенту:

`Codex, работай по AGENTS.md. Для существующего продукта выполни bootstrap и собери inventory из доступных источников. Смотри README кодовых репозиториев, CI-конфиги, OpenAPI/AsyncAPI/spec файлы, runbooks и wiki-документы. Сначала собери всё, что можешь подтвердить автоматически, потом покажи gaps и задай уточняющие вопросы по одному блоку.`

Если источники известны заранее, лучше сразу указать их явно:

`Codex, работай по AGENTS.md. Подними arch-docs для существующего продукта. Используй bootstrap. Источники: репозитории backend-platform, billing-api, customer-ui; OpenAPI лежит в backend-platform/openapi/; runbooks лежат в ops/runbooks/; старая wiki экспортирована в docs/legacy-wiki/. Сначала собери inventory из этих источников, потом покажи gap-report и задай вопросы по незакрытым местам.`

### 5. Не закрывать пробелы фантазией

Если ownership, фактический контракт или поведение среды не подтверждены, это нужно явно оставить как незакрытый gap.

Для зрелого продукта честный gap-report полезнее, чем красивая, но недостоверная документация.

При диалоге с агентом это означает простое правило: если факт неизвестен, отвечать `неизвестно` и дать агенту зафиксировать `Неизвестно / требует уточнения`.

## Как работать после инициализации

После того как репозиторий уже запущен и базово наполнен, он становится частью обычного change-management процесса.

### Типовой цикл изменения

1. Появляется новая фича, изменение поведения или архитектурный сдвиг.
2. Пользователь вызывает `analyst-architect` и через диалог получает draft feature-файла.
3. Пользователь вызывает `documentation-mapper`, и агент раскладывает impact по профильным разделам.
4. При необходимости агент обновляет контракты, схемы, deployment и тестовые сценарии.
5. Пользователь запускает review skills и валидирует замечания.
6. Изменения реализуются или сверяются с реализацией в кодовых репозиториях.
7. Пользователь вызывает `implementation-verifier` или обновляет release notes через agent workflow.
8. Агент синхронизирует `wiki-llm` индекс и прогоняет проверки.

### Какие действия делать почти на каждое изменение

- просить агента проверить, не затронуты ли требования;
- просить агента проверить, не изменился ли контракт API или события;
- просить агента проверить impact на данные и ownership;
- просить агента проверить test impact;
- просить агента проверить deployment/runtime impact;
- просить агента добавить или обновить release notes, если изменение пользовательски или операционно значимо.

### Как работать с Codex

Базовая дисциплина работы с Codex здесь такая:

1. Сначала сказать `Работай по AGENTS.md`.
2. Затем назвать конкретный workflow или skill под задачу.
3. Затем дать агенту право самому читать нужные файлы, задавать вопросы и вносить правки.
4. Затем валидировать итог через review skills и детерминированные проверки.

Плохой сценарий:

- пользователь сам редактирует десятки файлов;
- потом просит агента “проверить всё”;
- в итоге агент видит уже смешанный и частично недостоверный результат.

Хороший сценарий:

- пользователь формулирует задачу;
- агент ведёт пользователя по вопросам;
- агент сам меняет нужные документы;
- после этого агент или пользователь прогоняет валидацию результата.

Примеры рабочих запросов уже собраны в `.docs/codex-workflow.md`.

### Когда запускать review

Review skills стоит запускать:

- перед коммитом документации;
- перед согласованием крупной фичи;
- после заметных изменений контрактов;
- перед проверкой соответствия реализации документации.

Это тоже лучше делать через явный вызов агентов-reviewer, а не как абстрактную просьбу “посмотри, всё ли нормально”.

### Как работать с `wiki-llm`

`wiki-llm` нужен не для ручного редактирования, а как локальный retrieval-слой поверх документации.

Практически это значит следующее:

- агент опирается на `wiki-llm`, чтобы быстрее находить релевантные документы и не терять контекст;
- после изменений документации индекс нужно синхронизировать;
- если индекс не синхронизирован, агент может опираться на устаревшее состояние репозитория.

Что обычно делает пользователь:

1. Просит агента работать по `AGENTS.md`.
2. Даёт задачу: заполнить контур, описать фичу, разложить impact, провести review или verification.
3. После заметных изменений просит агента выполнить `wiki-llm sync` и затем `audit`.

Что обычно делает агент:

1. Читает `arch-docs.yaml` и проверяет, включён ли `wiki_llm`.
2. При необходимости обновляет индекс командой `./.arch-docs/scripts/wiki-llm-sync.sh`.
3. После синхронизации проверяет репозиторий через `./.arch-docs/scripts/audit-standard.sh --fail-on-gap`.
4. Продолжает работу уже с актуальным retrieval-контекстом.

Когда просить `wiki-llm sync`:

- после первичного bootstrap или заполнения нового репозитория;
- после заметных правок в `repository.yaml`, `overview.md`, `glossary.md`;
- после обновления feature-файлов и профильных разделов `architecture/`;
- перед review, если было много документационных правок;
- перед verification, если нужно сверять документацию с реализацией.

Хорошие запросы к агенту:

`Codex, работай по AGENTS.md. После правок синхронизируй wiki-llm и затем прогони audit.`

`Codex, работай по AGENTS.md. Сначала проверь, актуален ли wiki-llm индекс. Если нет, выполни wiki-llm sync, потом продолжай работу по фиче.`

Чего не нужно делать:

- не редактировать `.arch-docs/wiki-llm/index-manifest.yaml` вручную;
- не пытаться “заполнять wiki-llm” как отдельный документ;
- не пропускать sync после крупных изменений документации.

### Когда делать standard migration

Если меняется сам шаблон или организационный стандарт, это отдельный поток работы.

В этом случае нужно не редактировать `.arch-docs` вручную хаотично, а идти через:

```bash
./.arch-docs/scripts/audit-standard.sh --fail-on-gap
./.arch-docs/scripts/migrate-standard.sh
```

И использовать роль `standard-migrator` для анализа и применения обновления.

Хороший запрос к агенту:

`Codex, работай по AGENTS.md и используй skill .arch-docs/agents/standard-migrator. Сначала проверь расхождение с target_version, затем выполни migration workflow и после изменений прогони audit.`

## Практический минимальный operating model

Если упростить всю модель до рабочего минимума, то она выглядит так:

- `repository.yaml` отвечает на вопрос, из чего состоит система;
- `features/` отвечает на вопрос, какое изменение происходит;
- `architecture/` отвечает на вопрос, как система должна быть устроена;
- `release-notes/` отвечает на вопрос, что реально изменилось;
- `.arch-docs/` отвечает за локальный стандарт, проверки, automation и agent workflow;
- `.docs/` хранит обучающие и процессные материалы для людей.

Если команда поддерживает эти шесть слоёв в согласованном состоянии, инструмент работает как intended. Если хотя бы один слой выпадает, репозиторий быстро превращается либо в формальность, либо в архив устаревших файлов.

## С чего начать прямо сейчас

Для нового продукта:

1. Вызвать агент для заполнения системного контура.
2. Дать агенту закрыть placeholders и `unknown`, которые уже можно уточнить.
3. Попросить агента прогнать базовые проверки и довести репозиторий до рабочего состояния.
4. Вызвать `analyst-architect` для первой feature или capability.
5. После результата прогнать review и валидацию.

Стартовый запрос:

`Codex, работай по AGENTS.md. Помоги запустить новый arch-docs репозиторий end-to-end: сначала заполни системный контур через интервью, затем закрой известные placeholders, потом прогони базовые проверки, а после этого помоги описать первую capability через analyst-architect.`

Для существующего продукта:

1. Вызвать агент на bootstrap existing product.
2. Попросить его собрать inventory и gap-report из доступных артефактов.
3. Отвечать на точечные вопросы агента по ownership, интеграциям и средам.
4. Попросить агента итеративно поднять критичные разделы и прогнать проверки.
5. После этого вести все новые изменения через обычный feature-driven agent workflow.

Стартовый запрос:

`Codex, работай по AGENTS.md. Подними documentation repo для существующего продукта: собери as-is inventory, подготовь bootstrap gap-report, задавай уточняющие вопросы по одному блоку, обновляй документы и после каждой итерации валидируй результат.`
